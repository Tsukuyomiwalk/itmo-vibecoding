## Vision

### 10x Check

The 10x version is a Personal CFO in Telegram. The freelancer logs their incoming
payment (`/payment 2000 USD`), the bot watches rates, fires when favorable, logs
the conversion, and six months later tells them: "Average conversion premium: ₽0.8/USD
vs. day-of-receipt rate." The bot has memory, not just state. It becomes the answer
to "should I convert today?" — with personalized context, not generic price data.

### Platonic Ideal

A freelancer gets paid $2,000 USDT. They tell the bot. The bot watches USD/RUB,
fires when it hits 95 with context ("Highest in 14 days — good time"), the freelancer
taps one button to log the conversion, and the bot replies "₽190,000 received —
₽6,000 more than day-of-receipt rate." Six months later: personalized conversion
statistics. They show it to a friend. The friend adds the bot.

## Accepted Scope (added to this plan)

**APIs:** Fiat rates via open.er-api.com (primary). Crypto via CoinGecko free tier.
Both are already integrated in api_client.py.

**Base (from design doc / Approach B):**
- `/alert <code> <price> [above|below]` — threshold alerts (direction default: above)
  - Validation: code must be in KNOWN_FIAT_CODES or COINGECKO_IDS; price must be positive float
  - Behavior: case-insensitive code (`/alert usd 95` == `/alert USD 95`)
- `/alerts` — list active (untriggered) alerts: "USD: 93.5 (as of 14:55 UTC), target 95.0 (−1.6% away)"
- `/unalert <code>` — remove all untriggered alerts for that code
- APScheduler background loop (5-min check, in-process)
- `alerts` table with `was_triggered` idempotency flag (prevents double-fire on container restart)
- Cleanup job: delete triggered alerts older than 7 days (daily midnight UTC)

**Expansion 1 — Payment tracking:**
- `/payment <amount> <currency> [target_rate]` — log incoming payment
  - Validation: same as /alert. target_rate optional positive float.
  - Creates entry in `payments` table with status='pending'
  - If target_rate provided: auto-creates an alert (direction: always 'above' for payment alerts — favorable for RUB conversion)
  - If no target_rate: bot replies with current rate + "Add an alert? /alert USD 95"
  - Payment-auto-alert and manual /alert are independent rows in alerts table; both can exist for same code and both fire independently. No conflict, no deduplication.

**Payment state machine:**
```
  /payment added → status='pending'
  /converted called → status='converted', linked conversion record created
  /unwatch code → does NOT affect payments (payments are a separate flow)
```

**Expansion 2 — Conversion logging:**
- `/converted <amount> <currency> at <rate>` — log conversion
  - Example: `/converted 2000 USD at 94.5`
  - Validation: amount positive float, currency in known codes, rate positive float
  - Marks the oldest pending payment for that currency as 'converted' (FIFO by added_at)
  - Calculates P&L vs receipt-day rate: fetches rate at `payments.added_at` date from
    `rate_history` table; shows "₽189,000 received — ₽4,000 more than receipt-day rate"
  - If no pending payment for currency: just logs the conversion (standalone record)

**Expansion 3 — Weekly brief: DEFERRED to v1.1.**
Rationale: re-engagement is a valid goal but adds 2 new scheduler jobs, user_prefs table,
timezone complexity, and digest content logic. The on-demand `/alerts` command solves
the same check-in use case without the complexity. Ship alerts + payments + conversions
first; add digest after first real users provide feedback.

**Expansion 4 — Rate trend context in alerts:**
- When alert fires: "💱 USD/RUB hit 95.2\n↑ +1.8% vs yesterday's close\n📈 Highest in 14 days"
- Source: `rate_history` table — one row per code per UTC day (daily close = last recorded rate of the day)
- "Yesterday's change" = (today's rate - yesterday's close) / yesterday's close * 100
- "Highest in N days" = compare today's rate against last 14 rows for that code
- Populated by: APScheduler job writes to rate_history once per day at midnight UTC
  (daily close = last fetched rate of the day). Does NOT store intraday samples.
  5-min alert loop uses live API price directly — no overlap with rate_history reads.
- Day 0 initialization: on first check_alerts() run, seed rate_history with current rate
  for all codes present in active alerts (INSERT OR IGNORE by UNIQUE constraint).
- P&L fallback: if receipt-day rate not in rate_history (data gap), P&L field is NULL
  and the bot omits the P&L line from the confirmation message.
- 14-day window kept; rows older than 14 days purged by daily cleanup job (midnight UTC).
- APScheduler concurrency: SQLite write lock serializes access. Bot handlers and
  scheduler jobs share the same DB connection pool; concurrent writes are safe via
  Python's sqlite3 check_same_thread=False + connection-per-call pattern (already used).

**New tables needed:**
```sql
-- Payment tracking
CREATE TABLE payments (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id     INTEGER NOT NULL,
    amount      REAL NOT NULL,
    from_cur    TEXT NOT NULL,
    rate_at_add REAL,          -- rate at time of /payment command (from API)
    target_rate REAL,          -- optional, used to auto-create alert
    status      TEXT NOT NULL DEFAULT 'pending', -- 'pending' | 'converted'
    added_at    TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Conversion history
CREATE TABLE conversions (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id      INTEGER NOT NULL,
    amount       REAL NOT NULL,
    from_cur     TEXT NOT NULL,
    rate         REAL NOT NULL,  -- actual conversion rate
    rub_received REAL NOT NULL,  -- amount * rate
    pnl_rub      REAL,           -- vs receipt-day rate (nullable if no payment linked)
    payment_id   INTEGER,        -- FK to payments.id (nullable)
    converted_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Daily rate history for trend context
CREATE TABLE rate_history (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    code          TEXT NOT NULL,
    rate          REAL NOT NULL,   -- rate in RUB (1 USD = X RUB, etc.)
    recorded_date TEXT NOT NULL,   -- UTC date YYYY-MM-DD
    UNIQUE(code, recorded_date)    -- one row per code per day
);
```

## Reviewer Concerns (max iterations reached — not resolved in this document)

1. **Telegram user_id handling:** Assumed to come from `update.effective_user.id` (existing pattern in bot.py). Confirmed correct per Telegram Bot API — no session management needed.
2. **Database migration strategy:** No migration tooling. Pattern: `init_db()` uses `CREATE TABLE IF NOT EXISTS` — safe for new tables, not for column additions. Document known limitation: if user has existing bot.db, new tables are created automatically; no alembic/flyway for MVP.
3. **open.er-api.com free tier limits:** Not documented. Risk: if exceeded, fiat alerts silently fail until next cycle. Mitigation already in place: batch fetch reduces calls to O(1) per code per cycle. Monitor after real users.
4. **Multi-device / session handling:** Out of scope. Telegram handles identity via user_id — same across devices.
5. **User_prefs table in schema** (from weekly brief, now deferred): remove `user_prefs` from new tables list. Not needed for v1.
